# Vendor Comparison Brief

**Who this is for:** Procurement, IT, operations, PMs, founders, and teams comparing tools or service providers.

**Work problem:** Vendor comparisons often mix marketing claims, requirements, pricing notes, and stakeholder opinions. This workflow turns them into a source-backed comparison brief.

## Inputs

What you need before running this:

- Vendor list.
- Decision criteria and must-have requirements.
- Source links, notes, demos, proposals, or security documents.
- Constraints such as budget, timeline, data sensitivity, or integration needs.

## Copy-ready Codex instruction

```text
Create a vendor comparison brief from the inputs below.

Use only the source material I provide. Do not invent pricing, features, customer proof, security posture, integrations, or legal conclusions. If a vendor claim is unsupported, mark it as unsupported.

Decision:
[write]

Vendors:
[list]

Criteria:
[list must-haves and nice-to-haves]

Inputs:
[paste source links, proposal notes, demos, security notes, stakeholder comments]

Constraints:
[budget, deadline, compliance, systems, region, support needs]

Produce:
1. Recommendation summary: preferred option or "not enough evidence."
2. Criteria table: vendor, evidence, fit, risk, and unanswered questions.
3. Red flags: unsupported claims, missing docs, pricing uncertainty, data/security concerns.
4. Tradeoffs: what each vendor is best and worst for.
5. Next diligence steps: questions, documents, and approvals needed.
6. Source map: claim, source, date if provided, and confidence.

Keep sales language out of the memo.
```

## Expected output

A good run produces:

- A criteria-based vendor table.
- Clear unsupported-claim flags.
- Diligence questions for the next step.
- A recommendation only when the evidence supports one.

## Human review checklist

Before using the brief:

- [ ] Check vendor claims against source documents.
- [ ] Confirm pricing, contract, data processing, and security details.
- [ ] Validate criteria with stakeholders.
- [ ] Ask vendors to confirm missing or ambiguous information.
- [ ] Involve legal, security, procurement, or finance where required.

## Failure modes

Where this workflow can go wrong:

- It may over-weight vendors with more polished materials.
- It may treat marketing copy as evidence.
- It may miss procurement or security requirements not included in the input.
- It may recommend a vendor before decision criteria are stable.

## Source boundary

This workflow does not verify vendor claims or replace procurement, legal, security, or finance review. It organizes the material you provide into a decision draft.
